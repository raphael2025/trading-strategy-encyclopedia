## 三、回测层（Layer 4）

### 1. 回测引擎分类

**事件驱动回测**（Event-Driven）：
- 模拟交易所撮合
- 处理队列、延迟
- 准确但慢
- **代表**：Backtrader, Zipline, Lean

**向量化回测**（Vectorized）：
- 基于 numpy/pandas
- 速度快
- 不考虑订单簿、延迟
- **代表**：VectorBT, pandas-ta

**生产级回测**（Production-Grade）：
- 真实 tick 数据
- 滑点、佣金、手续费
- 清算模拟
- **代表**：QuantConnect, Lean Cloud

### 2. 关键回测陷阱

**未来数据（Look-ahead Bias）**：
- 使用尚未发生的数据
- 经典错误：用今日的收盘价做今日的交易
- **防御**：严格时间对齐

**过拟合（Overfitting）**：
- 参数过多、样本过少
- **防御**：样本外测试、Walk-Forward Analysis

**生存者偏差（Survivorship Bias）**：
- 只测试"现存"的币
- 忽略已下架/归零的币
- **防御**：使用历史全量数据

**交易摩擦忽略**：
- 不考虑手续费、滑点、资金费率
- 实战中可能让盈利策略变亏损
- **防御**：加入 0.05-0.10% 单边成本

**冲击成本忽略**：
- 大单对市价的冲击
- **防御**：用 VWAP/TWAP 模拟

**过度乐观的假设**：
- "完美填单"、"零延迟"
- **防御**：加入 100-500ms 延迟

### 3. Walk-Forward Analysis（WFA）

**步骤**：
1. 切分数据：Train 70% / Test 30%
2. 在 Train 上优化参数
3. 在 Test 上验证
4. 滚动窗口，重复

**加密应用**：
- 训练：2019-2022
- 验证：2023
- 训练：2020-2023
- 验证：2024

### 4. 性能评估指标

**收益指标**：
- **CAGR**（年化复合增长率）
- **Total Return**（总收益）

**风险指标**：
- **Sharpe Ratio**：超额收益/波动率
- **Sortino Ratio**：下行波动调整
- **Calmar Ratio**：收益/最大回撤
- **Max Drawdown**：最大回撤
- **Volatility**：年化波动率

**交易指标**：
- **Win Rate**：胜率
- **Profit Factor**：总盈利/总亏损
- **Avg Win / Avg Loss**：平均盈亏比
- **Number of Trades**：交易次数

**实战标准**：
- Sharpe > 1.5：合格
- Sharpe > 2.0：优秀
- Sharpe > 3.0：罕见（多半过拟合）
- Max Drawdown < 20%：良好
- Win Rate × Avg Win > Loss Rate × Avg Loss：期望值为正

## 四、策略层（Layer 5）

### 1. 策略分类

**按时间尺度**：

| 类型 | 持仓期 | 信号频率 | 技术栈 |
|------|--------|---------|--------|
| 高频（HFT） | 秒级 | 1000+/秒 | C++, FPGA |
| 中频 | 分钟-小时 | 10-100/天 | Python, C++ |
| 日内 | 日内 | 1-10/天 | Python |
| 中期 | 1-30 天 | 周/月 | Python |
| 长期 | 月-年 | 季度调仓 | Python |

**按信号类型**：
- **趋势跟踪**：均线、海龟、Donchian
- **均值回归**：配对、统计套利
- **动量**：Rank、涨幅
- **资金流**：订单流、链上数据
- **情绪**：社交媒体、新闻
- **事件驱动**：减半、ETF、FOMC

### 2. Alpha 模型

**动量 Alpha**：
- 过去 N 天收益率
- Rank IC: 0.05-0.10
- 加密市场显著

**价值 Alpha**：
- MVRV、NUPL
- 高 MVRV 卖出，低 MVRV 买入
- BTC 历史回测：年化 30%+

**资金流 Alpha**：
- 交易所净流入（卖出信号）
- 巨鲸动向
- 稳定币供应

**链上 Alpha**：
- 长期持有者 SOPR
- 矿工卖出压力
- 巨鲸钱包追踪

### 3. 组合构建

**简单等权**：每个资产 1/N
**风险平价（Risk Parity）**：每个资产贡献等量风险
**最大分散（Max Diversification）**
**Black-Litterman**：融合主观观点
**Kelly Criterion**：最大化长期增长率

**Kelly 公式**：
- 仓位 = (胜率 × 平均盈利 - 败率 × 平均亏损) / 平均盈利
- 例：胜率 60%，盈亏比 2 → 仓位 40%
- **实战**：通常用 1/4 Kelly 降低波动

### 4. 执行算法

**TWAP**（时间加权）：
- 在 N 个时间点平均执行
- 简单但易被检测

**VWAP**（成交量加权）：
- 按历史成交量分布执行
- 更自然

**Iceberg**（冰山）：
- 大单拆为小单
- 隐藏真实意图

**Sniper**（狙击）：
- 检测到目标价格立即市价
- 用于"猎杀"机会

**Implementation Shortfall**：
- 理论目标价 vs 实际执行价
- 最小化总成本

### 5. 滑点管理

**原因**：
- 市场冲击（大单推动价格）
- 时间延迟
- 信息泄露

**应对**：
- 用限价单替代市价单
- 拆分大单
- 选择流动性时段
- 用 TWAP/VWAP 算法
- 避开极端波动期

## 五、风险管理层（Layer 6）

### 1. 实时风控

**仓位控制**：
- 单资产 ≤ 5%
- 单策略 ≤ 20%
- 总杠杆 ≤ 2x

**止损**：
- 固定百分比（-2%）
- ATR 动态止损
- 时间止损

**最大回撤**：
- 触发 10% 回撤 → 减仓
- 触发 20% 回撤 → 停止交易

**保证金监控**：
- 维持保证金 ≥ 30%
- 强平预警

### 2. 投资组合风险

**VaR（Value at Risk）**：
- 95% 置信度下，1 天最大损失
- 例：VaR 5% → 100 万美元组合 1 天最大损失 5 万

**CVaR（Conditional VaR）**：
- 极端情况下的平均损失
- 比 VaR 更保守

**压力测试**：
- 2008 金融危机
- 2020-03-12 BTC 暴跌 50%
- 2022-05 Luna 崩盘
- 2022-11 FTX 暴雷

### 3. 模型风险

- 参数过拟合
- 数据问题
- 概念漂移（市场结构变化）

**防御**：
- 多策略组合
- 定期再训练
- 监控 IC 衰减

### 4. 操作风险

- 代码 bug
- 交易所 API 故障
- 网络延迟
- 人为错误

**防御**：
- 严格测试
- 监控告警
- Kill Switch
- 备份方案

## 八、加密量化特有问题

### 1. 24/7 市场

- **问题**：没有收盘价
- **应对**：用 4h/1d K 线作为"日线"
- **时区**：UTC 时间为标准

### 2. 高波动率

- **问题**：BTC 日均 3-5%，山寨币 10-15%
- **应对**：
  - 动态止损（ATR-based）
  - 降低杠杆
  - 仓位控制

### 3. 链上风险

- **桥接被黑**（Ronin 2022, Wormhole 2022）
- **合约漏洞**（多个 DeFi 协议）
- **预言机操纵**（多个案例）

### 4. 交易所风险

- **FTX 暴雷**（2022）
- **API 故障**（频繁）
- **提现限制**（极端行情）

**应对**：
- 多交易所分散
- 冷钱包存储
- 实时监控

### 5. 资金费率/借贷成本

- 永续合约的资金费率可达 ±0.1%/8h
- 长期持仓的隐性成本
- **应对**：监控 funding，合理选择工具

> 本附录为 06-01-框架 文档补充 6 个核心模块：① 配置管理 (Pydantic) ② 日志与监控 ③ 数据库 ORM (SQLAlchemy) ④ 事件总线 (ZeroMQ) ⑤ 部署 (Docker + K8s) ⑥ 完整项目骨架。

## A.1 配置管理 (Pydantic Settings)

```python
"""
类型安全的配置管理
- 支持 .env / YAML / 环境变量
- 自动验证
"""
from pydantic_settings import BaseSettings
from pydantic import Field
from typing import Literal
```

```python
"""
Loguru: 简单强大的日志
- 自动滚动、压缩、告警
- 集成 Sentry / Grafana

from loguru import logger
import sys


def setup_logging(log_dir: str = "./logs", level: str = "INFO"):
    logger.remove()
    logger.add(sys.stdout, level=level,
               format="<green>{time:YYYY-MM-DD HH:mm:ss}</green> | "
                      "<level>{level: <8}</level> | "
                      "<cyan>{name}</cyan> - "
                      "<level>{message}</level>")

               rotation="00:00", retention="30 days",
               compression="zip", level=level)
    return logger
```

## A.4 事件总线（ZeroMQ）

```python
"""
ZeroMQ: 进程间通信
- 低延迟，纳秒级
- 适合：行情 → 策略 → 执行 三进程解耦
"""
import zmq
import zmq.asyncio
import json


class EventBus:
    def __init__(self, port: int = 5555):
        self.ctx = zmq.asyncio.Context()
        self.socket = self.ctx.socket(zmq.PUB)
        self.socket.bind(f"tcp://*:{port}")

    async def publish(self, topic: str, data: dict):
        await self.socket.send_multipart([
            topic.encode(),
            json.dumps(data, default=str).encode()
        ])


class EventSubscriber:
    def __init__(self, port: int = 5555, topic: str = ""):
        self.ctx = zmq.asyncio.Context()
        self.socket = self.ctx.socket(zmq.SUB)
        self.socket.connect(f"tcp://localhost:{port}")
        self.socket.setsockopt_string(zmq.SUBSCRIBE, topic)

    async def listen(self):
        while True:
            topic, data = await self.socket.recv_multipart()
            yield topic.decode(), json.loads(data)
```

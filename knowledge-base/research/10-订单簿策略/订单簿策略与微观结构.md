## 一、订单簿基础与 L2/L3 数据

### 1.1 订单簿 (Limit Order Book, LOB) 本质

订单簿是连续双向拍卖的物理实现：每个价格水平 (price level) 上挂有买/卖两个方向的限价单队列 (queue)，未成交的市价单按价格优先-时间优先 (price-time priority) 在队列里撮合 [[Rama Cont Wikipedia](https://en.wikipedia.org/wiki/Rama_Cont)]。在加密市场，主流交易所 (Binance、OKX、Bybit、Coinbase、Coinbase Advanced) 都采用中央限价订单簿 (Central Limit Order Book, CLOB)，但匹配引擎有差异：

- **Binance Spot** 撮合频率标称 100 万次/秒，单 symbol 盘口深度 5000 档（来自 `depth20` / `depth50` WebSocket 流；`depth` 推送上限 5000 档，由 `<symbol>@depth<levels>@<speed>` 订阅指定）[[Binance WS Streams](https://developers.binance.com/docs/binance-spot-api-docs/web-socket-streams)]。
- **OKX** 现货 5 档/20 档/`books` 400 档推送，频道 `books5` / `books50-l2-tbt` (tick-by-tick 真实增量)，提供 trade-by-trade (`trades`) 与 `funding` 频率 8 小时推送 [[OKX v5 WebSocket](https://www.okx.com/docs-v5/en/)]。
- **Bybit** Spot v5 文档披露 `orderbook.50` / `orderbook.200` / `orderbook.500` 三个深度档位，linear 与 inverse 衍生品有独立频道 `orderbook.200`；现货仅 depth-50、depth-200，对照 Binance、OKX 覆盖稍浅 [[Bybit v5](https://bybit-exchange.github.io/docs/v5/intro)]。
- **Coinbase Advanced Trade** 公开 L2 频道 `level2` 仅 50 档，但提供 `ticker` 增量；Coinbase Exchange (前 Pro) 旧 `full` 频道 5000 档已逐步下线，当前主推 Advanced Trade REST `/api/v3/brokerage/market/product_book` 支持 `limit=50` 参数 [[Coinbase CDP](https://docs.cdp.coinbase.com/)]。

### 1.2 L2 vs L3 关键差异

- **L2 (Level 2 / 价格聚合数据)**：只显示价格档位和该档位上的 *总量*，不暴露挂单数量细节、订单 ID 或挂单时间。常见字段：`bids[price, qty]`、`asks[price, qty]`。绝大多数公开 API 推送的是 L2。
- **L3 (Level 3 / 订单明细数据)**：暴露每个挂单的 order_id、原始数量、挂单时间戳、修改/取消日志。NASDAQ、NYSE、CME 都对机构直接连接 (Sponsoring Member) 提供 L3 内部数据；加密交易所中 Binance、OKX、Bybit 因风控需求不公开 L3，但**订单流日志 (L3-replay)** 会被内部撮合引擎记录。

学术文献 (Cont, Kukanov, Stoikov 2014, "Price Dynamics in Limit Order Markets"; 后被 Cont 2021 的 arXiv 2006.09643 综述引用) 通过对比 L2 与 L3 发现：L2 隐含队列长度会低估 hidden iceberg 订单与 cancel-replace 行为的信号；L3 重建的 *cancel rate* 是 L2 数据上 *quantity-weighted mid change* 的 1.7 倍预测能力 (in-sample R²) [[Cont 2021 arXiv 2006.09643](https://arxiv.org/abs/2006.09643)]。

### 1.3 盘口快照 vs 增量更新

WebSocket 推送的两种核心形态：

| 形态 | 字段 | 优点 | 缺点 |
|---|---|---|---|
| **Partial Book (快照)** | `bids`、`asks` 完整 K 档 | 客户端无需维护本地状态 | 流量大、UDP-like, 50 档≈ 2 KB / 推送, BTC 主流交易所推送频率 100ms-1s |
| **Delta / Top-of-Book (增量)** | 单条 diff: `(price, new_qty, side)` | 带宽节省 10x | 必须按 `(lastUpdateId, prevUpdateId)` 严格排序，丢包需触发 REST `/depth` 重同步 |

Binance Spot `@depth` 频道约定：若 `(u, U)` 序列断裂，必须先 GET `/api/v3/depth?limit=5000` 重新对齐 `lastUpdateId`；OKX `books-l2-tbt` 用 `prevSeqId` / `seqId`；Bybit `orderbook.200` 用 `u` (update id) 连续；Coinbase Advanced `level2` 公开的 `updates` 事件流附带 `time` 字段 [[Binance WS Streams](https://developers.binance.com/docs/binance-spot-api-docs/web-socket-streams)]。

### 1.4 合成 L3 / 推断 L3

虽然 L3 原始数据被封闭，但研究者常用以下方法从 L2 增量流 + trade 流向"合成 L3"：

1. **Order-to-Trade Ratio (OTR) 反推**：每个主动成交单 (taker trade) 必然吃掉一个最优 L2 档位的 L3 单；若同 price 出现 cancel 但 qty 减少不匹配 trade，可推断"挂单者自撤"；若 trade qty 超过最优档位，则推断队列被吃穿 (queue jump) 并触发 *Level Fill* 事件。
2. **Queue-Reconstruction** (Cartea, Jaimungal, Penalva 2015, "Algorithmic and High-Frequency Trading")：维护 `in_queue_qty[price][order_id_or_hash]`，每个 L2 delta 视为单笔 L3 增/删。
3. **L3-Tape-Clustering** (Bouchaud 等)：将 trade 流的 price-qty 序列按 z-score 分簇，识别出隐藏 icebergs 的循环补单模式。

> **工程陷阱**：L3 重建属于 *强假设*，不要把"推断队列长度"当作 ground truth 喂给 ML 模型，应该作为特征工程的辅助输入。盲目训练会把 *modeling artifact* 当 *alpha*。

### 小结 (一)

订单簿是所有高频/做市策略的物理底层。L2 数据是行业标准，但只有知道每个交易所推送的 (a) 档位深度 (b) 增量协议 (`lastUpdateId`/`seqId`) (c) 重同步机制，才能写出不会丢包/失序的撮合引擎。L3 数据不公开，但通过 L2 增量 + trade 流的 *order-flow disaggregation* 可以近似恢复单笔订单的进出时机，学术界 (Cont 2014) 和 Cartea-Jaimungal-Penalva (2015) 的专著都讨论过这条路径。

## 二、订单簿不平衡信号 (OBI)

### 2.1 经典 OBI 公式

订单簿不平衡 (Order Book Imbalance, OBI) 是衡量盘口买卖力量不对称的标量，原始定义 (Kaul, Nimalendran 1990, Chordia-Roll-Subrahmanyam 2002)：

$$
\text{OBI}_k = \frac{Q^{bid}_k - Q^{ask}_k}{Q^{bid}_k + Q^{ask}_k}
$$

其中 $Q^{bid}_k$ 与 $Q^{ask}_k$ 是 top-$k$ 档买方/卖方累计挂单量。范围 $[-1, +1]$，正数代表买盘强势。Cont (2021) 在 *Empirical properties of order flow in financial markets* (arXiv 2006.09643) 中系统化了 OBI 的预测能力：对 BTC-USDT 1 秒后 mid-price change，OBI$_1$ 与 $R^2$ 的 in-sample 拟合度在 0.5%-1.5%，跨交易所、跨时段稳健 [[Cont 2021](https://arxiv.org/abs/2006.09643)]。

### 2.2 多档加权 OBI 与 Volume-Weighted OBI

直接对多档求和会过度重视远离最优档的"幽灵流动性" (phantom liquidity)，因为远端挂单被 cancel 的概率远高于最优档 (Bouchaud 等测算 80% 的限价单在 5 秒内被撤)。常用变体：

- **Exponential Decay OBI**：$\text{OBI}_\lambda = \sum_{i=1}^{K} e^{-(i-1)/\lambda} \frac{q^b_i - q^a_i}{q^b_i + q^a_i}$，其中 $\lambda$ 控制衰减速率；BTC 主流币 $\lambda \in [1, 5]$，以太坊 $\lambda$ 通常更小 (因为盘口薄)。
- **Volume-Weighted Mid (VW-Mid)**：$\text{mid}^{vw} = \frac{Q^a_1 \cdot P^b_1 + Q^b_1 \cdot P^a_1}{Q^a_1 + Q^b_1}$，隐含"挂单多的一侧"价格权重更高。
- **Microprice (Stoikov 2018)**：$p^{\text{micro}}_t = P^b_1 \cdot \frac{Q^a_1}{Q^a_1 + Q^b_1} + P^a_1 \cdot \frac{Q^b_1}{Q^a_1 + Q^b_1}$，数学上等价于 VW-Mid 的 mirror，含义是 *expected mid next tick*。

> **学术说明**：Cont, Kukanov, Stoikov (2014, "Price Dynamics in Limit Order Markets") 用 NYSE TAQ 数据显示 microprice 对 1-tick mid change 的 $R^2$ 约为 1.7%，比 naive mid 提升 1.4 倍；后续被 Cont 2021 (arXiv 2006.09643) 综述 [[Cont 2021](https://arxiv.org/abs/2006.09643)]。

### 2.3 加密适配与阈值经验

加密盘口行为与传统股票 LOB 有三大差异，必须在阈值上调整：

1. **Cancel-to-Trade Ratio (CTR) 高**：Binance BTC-USDT 现货的 1 秒内 cancel 量 / trade 量 长期在 30-100 倍区间，远高于 NYSE ECN 的 5-10 倍。这意味着 OBI 的 *瞬时* 值不可信，必须用 EWMA 平滑 ($t_{1/2} = 1\text{-}5\text{s}$)。
2. **Quote Stuffing 频发**：Maker 通过瞬时插撤订单制造伪 OBI 极端值，2019-2020 Kraken 和 Binance 都出现过。需用 z-score filter（窗口 60s）剔除 $|z| > 4$ 异常点。
3. **跨所领先-滞后**：Binance 流动性最深，OKX、Bybit 同步滞后 200-800ms；Coinbase 现货 (美东时间) 与 Binance Asia 时段同步性差。

**实战阈值** (BTC 现货, 经验值，非投资建议)：
- OBI$_1$ > +0.30 持续 5 秒以上 → 多头偏向信号
- OBI$_1$ < -0.30 持续 5 秒以上 → 空头偏向信号
- OBI$_5$ 与 OBI$_1$ *反向* (OBI$_1$ = +0.4 但 OBI$_5$ = +0.05) → 假突破概率高，是 maker 的伪 depth

### 2.4 衍生：Spread Skew, Depth Ratio, Book Pressure

- **Spread Skew** = $(P^a_1 - P^b_1 - s) / s$，其中 $s$ 是 quoted half-spread，反映 *taker* 的瞬时冲击成本预期。
- **Depth Ratio** = $Q^a_1 / Q^b_1$，相比 OBI 对 noise 更稳健 (因为没有中心化归一化分母为零的问题)。
- **Book Pressure (Bouchaud)**：$\text{Press}_t = \int_0^{\infty} \bigl(Q^b(p) - Q^a(p)\bigr) dp$，对全档位做积分；是 *物理压力* 的直观类比。

### 小结 (二)

OBI 是订单簿策略里**最便宜、最稳健**的 alpha 之一。核心思想是用 *最优档 + 第二档* 的挂单量预测 *下一笔 mid*，但切忌把瞬时 OBI 当信号；加密盘口的高 CTR 让 cancel 成了虚假 depth 的来源。Cont 2021 (arXiv 2006.09643) 与 Cont-Kukanov-Stoikov 2014 (SSRN 2402703) 是该领域最权威的引用基准，Cartea-Jaimungal-Penalva 2015 的教科书对理论进行了完整公式化。

## 三、订单流不平衡 (OFI) 与市场冲击

### 3.1 Order Flow Imbalance (Cont 2021)

Cont (2021) 在 arXiv 2006.09643 把 OFI 定义为 *单位时间内订单簿顶档 supply/demand 的净变化*，区别于 OBI (静态快照)：

$$
\text{OFI}_n = \Delta P^b_n \cdot \mathbb{1}_{\Delta P^b_n > 0} - \Delta P^a_n \cdot \mathbb{1}_{\Delta P^a_n > 0} + \Delta Q^b_n \cdot \mathbb{1}_{\Delta P^b_n = 0} - \Delta Q^a_n \cdot \mathbb{1}_{\Delta P^a_n = 0}
$$

其中 $P^b_n, Q^b_n$ 分别是第 $n$ 时刻买一档价格与数量；$\Delta P^b_n = P^b_n - P^b_{n-1}$，价格上升视为新增需求 ($\mathbb{1}_{\Delta P > 0} = 1$)，价格不变但 qty 增加视为新挂单。论文用 2014 年 NYSE TAQ 528 只股票回测，OFI 对 1-tick mid-change 的 $R^2$ 中位数 8.6%（远高于 OBI 的 1-2%）；当以 *signed order flow* (多档位 OFI 加权) 计算时，$R^2$ 可达 14-25% [[Cont 2021](https://arxiv.org/abs/2006.09643)]。

**加密适配** (BTC 现货, 1s 窗口)：
- 多档 OFI 优于 single-level OFI；
- 加入"cancel 权重"后, $R^2$ 提升 0.3-0.7 个百分点；
- 与"large trade"过滤叠加后，对 5 秒后 mid 预测的 sharpe 信号 ratio 显著。

### 3.2 Kyle's Lambda (1985)

Kyle (1985) 在 "Continuous Auctions and Insider Trading" (Econometrica 53(6)) 中提出 *price impact 模型*，把价格变化分解为信息 + 流动性噪声 + market impact：

$$
\Delta P_t = \lambda \cdot \text{sign}(X_t) \cdot \sqrt{|X_t|} + \epsilon_t
$$

其中 $X_t$ 是带符号的 net order flow，$\lambda$ (Kyle's lambda) 代表"每个市场参与者的信息敏感度"；它与市场流动性成反比。加密盘口的 $\lambda$ 估计：

- **Regressed Lambda** (Roll 1984 简化版)：用 1-min candle 回归 $\Delta P \sim \sqrt{V}$，斜率即 $\lambda$。BTC 现货 $\lambda$ ≈ 0.05-0.20 bps / √(BTC)；ETH 现货 $\lambda$ 略高 0.10-0.40 bps / √(ETH)。
- **实际意义**：做市商需要为每交易 1 BTC 支付约 5-20 bps 的 *taker cost*；1 ETH 约 10-40 bps。这是 design spread 的下限。

### 3.3 Amihud Illiquidity Ratio (2002)

Amihud (2002, "Illiquidity and stock returns") 的 illiquidity measure 简化版：

$$
\text{ILLIQ}_t = \frac{|R_t|}{V_t}
$$

其中 $R_t$ 是回报率 (绝对值), $V_t$ 是成交额 (USD)。在加密场景：

- BTC 1-min ILLIQ ≈ $1 \times 10^{-9}$ / USD
- 小币种 (e.g., PEPE) ILLIQ 可能高 100x-10000x，意味着同等 volume 下价格冲击大 100-10000 倍。

**实战用法**：Amihud ratio 是 *risk budgeting* 的天然输入；可以按 $1/\sqrt{\text{ILLIQ}}$ 缩放仓位，对小币种自动降权。

### 3.4 Spread Decomposition (Glosten-Harris 1988, Huang-Stoll 1997)

报价价差可以被分解成三部分：
- **Adverse Selection (信息成本)**：占加密现货 30-60%
- **Inventory Holding Cost**：占 10-30%
- **Order Processing Cost (maker rebate)**：占 20-50% (Binance、OKX、Bybit 都有 maker rebate 或 zero-fee 期)

Glosten-Harris 模型：
$$
\Delta P_t = \theta \cdot \Delta x_t + \psi \cdot x_t + \epsilon_t
$$

其中 $x_t$ 是累计 signed trade，$\theta$ 是 *permanent* price impact (信息成本)，$\psi$ 是 *transient* impact (inventory)。

### 3.5 Trade Size Power Law (Gabaix 2003)

Gabaix 等 2003 (arXiv cond-mat/0501292 与 Nature 2003) 发现 *trade size distribution* 服从 $P(\text{size} > s) \sim s^{-\alpha}$ 的 *power law*，$\alpha \approx 1.5$ 在股票市场；在加密市场更接近 1.0-1.4 (即更重的尾部)。这意味着：

- 极端大单虽然稀有但 *expected size* 远高于股票市场；
- 盘口深度在 *常态* 下看像 OK，但面对 $>\$1M$ 的 taker 单时 *价格弹性* (price impact 弹性) 显著高于股票。
- 实战中：BTC 现货 \$1M taker 单的 *slippage* 中位数约 1-3 bps (Binance BTC-USDT 主流时段)，$5M taker 单约 5-15 bps，$20M+ taker 单可能 30-100 bps。Glosten-Harris / Almgren-Chriss 模型对滑点的拟合较好 [[arXiv cond-mat/0501292](https://arxiv.org/abs/cond-mat/0501292)]。

### 小结 (三)

OFI 是订单簿策略的"动态版本"——比 OBI 多了一个 *时间维度*。Cont 2021 的实验显示，OFI 1-tick mid 的 $R^2$ 比 OBI 高 5-10 倍，但这依赖 *多档位加权 + cancel 修正*。Kyle's lambda / Amihud illiquidity / Glosten-Harris spread decomposition 共同构成 *price impact* 的三个互补度量，是做市与执行算法的定价基础。

## 四、流动性供给指标

### 4.1 PIN (Easley-O'Hara 1992)

PIN (Probability of Informed Trading) 是经典 *information-based* 流动性指标。模型假设每笔 trade 来自 *informed* 概率 $\alpha$ 或 *uninformed* 概率 $1-\alpha$；日内买卖单到达率 $\epsilon_b, \epsilon_s$ 由 $\alpha$ 决定；用极大似然估计 $\alpha$：

$$
\text{PIN} = \frac{\alpha \mu}{\alpha \mu + (1-\alpha)(\epsilon_b + \epsilon_s)}
$$

$\mu$ 是 informed 的到达率，PIN 越大，知情交易占比越高，对应 *逆选择成本* 越大。**加密市场痛点**：PIN 估计对样本量敏感，日内交易数 $N < 100$ 时方差爆炸；BTC 现货日交易数 $10^5$-$10^6$，估计尚可；小币种日 $10^3$ 量级，PIN 几乎不可用。

### 4.2 VPIN (Easley-López de Prado-O'Hara 2011)

Volume-Synchronized PIN (VPIN) 把 trade 按 *固定 volume 桶* 分组（典型 $N = 50$ 桶），计算每桶的 $|Buy - Sell|$ 之和；用 cumulative sum / total volume 得到 *toxicity flow*。原论文 (SSRN 1362468, *Flow Toxicity and Liquidity in a High-frequency World*) 用 S&P 500 E-mini 期货做 flash crash 检测，VPIN 在 2010-05-06 flash crash 前 1 小时 *先行突破阈值*。

**加密实战**：
- 选 50 桶 BTC-USDT 1-min volume，估算 *bucket size* ≈ \$3-10M；
- VPIN > 0.7 持续 5 min → 高 toxic flow 警告；
- 实战中 VPIN 在 Binance 现货上对 *short-term reversal* 信号更有效 (5-15 min 窗口)。

### 4.3 EHOO / PIN-L 改进

EHOO (Easley-Hardouvelis-O'Hara 2010, *Leaning Against the Wind*) 把 PIN 拓展为时变版本；PIN-L (Duarte-Lo 2008) 用 latent state 替代稳态假设。在加密盘口，时变 PIN 仍是研究前沿。

### 4.4 Queue Position / Time-Weighted Spread

**Queue Position Model (QPM, Stoikov 2018, *The micro-price*)**：做市商的 *position in queue* 决定成交概率。最优档挂单 $q^b$ 总量为 $Q^b_1$，做市商挂单量为 $q^b_m$，则下一笔 market sell 吃掉该做市商单的概率：

$$
P(\text{fill}) = \frac{q^b_m}{Q^b_1}
$$

如果做市商挂在 *top*，先吃完的 *不是* 自己，但要等前面 $Q^b_1 - q^b_m$ 全部成交。**时间加权 spread** = 实际 spread / 成交等待时间，是衡量 *active* spread 的核心指标。

### 4.5 Depth Decay & Refill (Liquidity Regeneration)

订单簿深度 (depth) 随时间 *decay*，被吃掉后会 *refill*：
- **Half-life of depth**：BTC 现货买一档 depth 的中位 half-life ≈ 5-15 秒；做市商撤单与新 maker 加入的平衡。
- **Refill Latency**：在 *吃单* 后 < 100ms 内通常有 maker 补单；空窗期 (< 50ms) 是 *亚秒级* 策略的 alpha 来源。

### 4.6 Cross-Venue Spread & Triangular Arbitrage

跨交易所 spread：$\text{spread}^{xy} = \log P^x_t - \log P^y_t$，扣除 funding / 提现费 / 排队延迟后，理论套利空间 1-10 bps。BTC-USDT 主流时段 Binance / OKX / Bybit / Coinbase 4 站点的 spread 标准差 < 5 bps。

三角套利 (triangular arb)：在 BTC-USDT, ETH-USDT, ETH-BTC 三个 pair 之间捕捉 $\log P_{\text{BTC-USDT}} - \log P_{\text{ETH-BTC}} - \log P_{\text{ETH-USDT}}$ 的偏零信号。Quadriga 等已经关闭的小所显示，三角套利窗口期可被吃光 50% within 200ms。

### 小结 (四)

流动性供给指标 (PIN, VPIN, EHOO, QPM, depth decay) 是做市与监控的 *信号库*。加密市场因为数据噪声大，需要 *rolling window + EWMA* 双重平滑；cross-venue spread 是低风险套利策略的直接输入。在实践上，VPIN + Queue Position + Refill Latency 是 *做市策略 alpha* 的三大主流组合。

## 五、做市策略 (Market Making)

### 5.1 Avellaneda-Stoikov (2008) 经典模型

Avellaneda 与 Stoikov 2008 在 NYU 数学系工作论文 *High Frequency Trading in a Limit Order Book* 中提出 *single-asset 做市* 的 stochastic control 框架 [[Avellaneda-Stoikov 2008 (Wayback Machine mirror)](https://web.archive.org/web/2018/math.nyu.edu/~avellane/HighfrequencyTrading.pdf)]。

**核心方程**：在 mid-price $S_t$ 服从均值回归 + 方差 $\sigma^2$ 的情况下，最优 bid / ask 报价相对 mid 的偏移：

$$
\delta^a_t = \delta^b_t = \frac{\sigma \sqrt{T - t}}{2} + \frac{\gamma \sigma^2 (T - t)}{\kappa}
$$

加上 inventory penalty：

$$
\text{bid}_t = S_t - \frac{q_t \gamma \sigma^2 (T - t)}{2} - \delta^a_t, \quad
\text{ask}_t = S_t - \frac{q_t \gamma \sigma^2 (T - t)}{2} + \delta^b_t
$$

其中：
- $q_t$ 是 inventory (正 = 多头)
- $\gamma$ 是 risk aversion
- $\kappa$ 是 order book 流动性参数 (价格弹性倒数)
- $T - t$ 是 horizon

**核心洞察**：当 $q > 0$ (持有库存)，$\text{bid}_t$ 抬高、$\text{ask}_t$ 压低 → 鼓励卖出；$q < 0$ 反之。这是 inventory 偏置 (skew) 的数学基础。

**加密适配问题**：
- 假设 *mid 是均值回归*——BTC 不是！需要把 *drift* $\mu$ 显式加进方程；
- *constant volatility* 假设对加密不成立，$\sigma_t$ 需用 GARCH 或 RV 实时估计；
- *single asset* 假设失败：BTC 现货 + 永续 + 季度期货是 *correlated* 三资产问题；
- *order arrival 是 Poisson* 假设对加密高 CTR 也不成立。

### 5.2 GLFT (Guéant-Lehalle-Fernandez-Tapia 2013)

Guéant, Lehalle-Fernandez-Tapia (arXiv 1305.0463, 2013, *Dealing with the inventory risk: a solution to the market making problem*) 是 Avellaneda-Stoikov 的 *closed-form 实用版*：

- 把 inventory 风险管理内生化为 *quote skew*；
- 给出 4 个具体参数 (`γ` risk aversion, `A` liquidity, `k` order arrival)；
- 在多个欧洲股票市场 backtest 提升 PnL 10-30% (相对 vanilla 策略) [[GLFT 2013](https://arxiv.org/abs/1305.0463)]。

**实战公式**：

$$
\text{bid}_t = S_t - \frac{1}{\kappa} \ln\left(1 + \frac{\gamma \kappa \sigma^2 (T - t)}{2 \text{ask\_size}}\right) - \frac{q_t \gamma \sigma^2 (T - t)}{4}
$$

GLFT 的优点是 *spread 与 skew 解耦*，方便在线性参数空间做 optimization；缺点是仍然 single-asset 假设。

### 5.3 库存风险与 Adverse Selection

**Adverse Selection (逆选择)**：
- 静态 PnL = $\sum \text{spread} - \text{slippage} - \text{adverse selection cost}$
- 逆选择成本 = 做市商被 informed flow 吃掉的亏损，估测占 spread 的 30-60%
- 衡量：mark-out PnL (Mark-out P&L)——成交后 $k$-ms 的 mid movement 减去 spread 的一半；负值越大，逆选择越严重
- 实战中 mark-out 5s 长期 < 0 → 做市商"被打"

**Inventory Risk (库存风险)**：
- 库存绝对值 $|q| > q_{\max}$ → 触发 *flatten* 强制减仓
- *drawdown* 监测：5-min rolling max drawdown > 2σ → 暂停做市
- 加密市场 BTC 现货做市的 *标准 inventory limit* 约 2-5 BTC (中等 size)；>10 BTC 即是 *big*。

### 5.4 四价 vs 两价 (Four-Quote vs Two-Quote)

**两价 (传统)**：bid-1, ask-1。
**四价 (4-quote / ladder)**：在 top-1 之外同时在 top-2, top-3 挂单：
- 优点：捕获"价格快速穿越"事件；提供"亚档"流动性赚 spread
- 缺点：增加 *adverse selection*——top-2 距离 mid 更远，toxic flow 跳过去的概率更高
- 4-quote *只适用于* 高波动 + 强 trend 时段；trending 行情下应该 *reduce* 远端 quote 的 size

**Staged Ladder (Hardy 2023, "Electronic and Algorithmic Trading")**：用 5-10 档不同 size 的 ladder，分层 *inverted* / *pyramid* 形态，配合 *cancel-on-touch* (CoT) 规则。

### 5.5 永续做市与 Funding Rate 套利

永续合约 (perpetual futures) 引入 *funding rate*：每 8 小时 (Binance、OKX、Bybit) 一次，资金费由 long 付给 short (当 funding > 0)。**做市商机会**：

1. **Delta-Neutral Funding Capture**：现货做空 (或持有反向 perp) 同时做多 perp 收取 funding，理论上无方向风险。BTC funding 在牛市中常 10-30% APR，在熊市 -10% 到 -30%。Funding capture 是 *机构级* 主要策略之一。
2. **Funding Rate Skew**：当 perp 显著 *溢价* 于 spot (basis > 0)，做市商可以 *short perp + long spot* 锁定期现套利；basis 在加密牛市中常达 10-50% APR。

> **关键风险**：*Liquidation cascade*——当 perp 暴跌，long 仓爆仓时 *auto-deleveraging* 机制会触发，spot / perp 价差瞬间打穿；2024-08-05 全球 flash crash 期间大量 delta-neutral 仓位被强平 [[Federal Reserve 2024-10 FEDS Notes](https://www.federalreserve.gov/)]。

### 5.6 机构做市商: Citadel Securities / Jane Street / Jump Crypto / Wintermute

四家顶级机构在加密做市市场占据主要份额 (Kaiko 2024 估算 70%+ 的 BTC-USDT 现货 maker volume)：

- **Citadel Securities** 自 2022 年起在 Binance、Coinbase、Kraken 提供 BTC/ETH 现货做市；据 SEC 2023 申请成为 ATS / 经纪自营商；其 *Citadel Connect* 是 RFQ 撮合平台。
- **Jane Street** 通过 Jane Street Digital 进入加密，2023 年在 Kraken、Bitstamp 提供现货 + 衍生品做市；2019 年起投资 NEAR、Flow 等 L1 链。
- **Jump Crypto** 是加密原生做市商，覆盖 BTC / ETH 主流 + 100+ altcoins；2022-11 因 FTX 倒闭损失约 \$300M，是 *机构风险* 案例。
- **Wintermute** 是加密 OTC + 现货做市领头羊，2022-06 被 Wintermute 攻击损失约 \$160M，凸显 *智能合约 / 钱包密钥* 风险。

### 5.7 失败案例：Alameda Research 2022 倒闭

Alameda Research 是 SBF (Sam Bankman-Fried) 旗下量化 + 做市机构。2022-11 FTX 崩盘后披露：
- 持有大量 *FTT* (FTX 平台币) 作为 collateral；
- 在 *流动性* 较差的 *小币种* 集中持仓 (SRM, MAPS, OXY 等)；
- 与 FTX 共享 *客户资金*，最终 *leverage* 极度高；
- 2022-11-09 申请破产；FTX 客户损失数十亿美元 [[Alameda Research Wikipedia](https://en.wikipedia.org/wiki/Alameda_Research)]。

**对做市策略的教训**：
1. *Concentration risk*——做市商不应在自家相关 token 上有 large inventory；
2. *Related-party transactions* 必须严格隔离；
3. *Real-time risk monitoring* 不能依赖单一 counter-party 的 solvency；
4. *Smart contract / 私钥* 安全是 100% 自留的 operational risk。

### 小结 (五)

做市是订单簿策略的 *最纯粹* 形式。Avellaneda-Stoikov 与 GLFT 给出理论框架；实践中需要 (a) 实时估计波动率 (b) inventory 偏置 (c) 4-quote 进阶 (d) 跨资产 funding 套利。机构做市商有 4 大头部 (Citadel Securities, Jane Street, Jump Crypto, Wintermute)；失败案例 (Alameda) 提醒 *单点故障* 与 *集中持仓* 的毁灭性。

## 六、激进型策略 (Aggressive / Taker Strategies)

### 6.1 Momentum Ignition (动量点火)

Momentum Ignition 是 *人为制造 trend* 的操纵手段 (CFTC 多次定义)；但合法的 *trend-following* 策略也利用相同信号：
- 监测 *OBI 急剧 > 0.5* + *OFI surge* + *taker volume spike* 三连击；
- 用 limit order *chase* 涨势 (post-only limit with cancel-on-touch)；
- 在 *first 200ms* 内建仓完毕，*目标 hold 1-5min*；
- 风险：信号可能是 *spoof*，3-5 秒后反转。

### 6.2 Iceberg 探测 (Hidden Order Detection)

Hidden / Iceberg 订单的特征是 *循环补单*：每隔 $\Delta t$ 出现一次大小相近的 limit order。检测方法：
1. **Time-clustering**：对同一 price level 的 L2 delta 用 DBSCAN 或 k-means 聚类，发现 *周期性* 模式；
2. **Volume-signature**：冰山单成交量是 *deterministic* 的 (固定 child size)，而主动交易是 *stochastic*；
3. **Reveal Probability**：当冰山单露出后，市场有 *知情 follow-up*，价格快速向该方向移动；可以利用 Bayesian 方法实时估计 *hidden size*。

> **法律警示**：利用 iceberg 信息在 *T+0* 内进行掠夺性交易，在美国受 Reg NMS Rule 611 限制；在加密盘口，*bid-ask* 内外都是 *single venue*，但跨所操作仍需评估监管。

### 6.3 Queue Jumping / Top-of-Book Sniping

在最优档前面"插队"：
- **Queue jumping**：用 sub-1ms 延迟抢在新 maker 前面挂单；
- **Top-of-book sniping**：当 *signal* 出现，立刻用 IOC (Immediate or Cancel) 吃掉对方的 top-1；典型场景：捕捉 *funding flip* 或 *现货 <-> 永续* basis 错位。
- 在 Binance / OKX 这种 *matching priority* 严格按 *price-time* 的场所，**唯一**抢在前面的方法是 *更低的延迟*——因此 co-location 是 alpha 的物理壁垒 [[Co-location Wikipedia](https://en.wikipedia.org/wiki/Co-location)]。

### 6.4 跨所 + 同所 Co-located Latency Arb

**同所 (Intra-venue) Co-located Latency Arb**：
- Binance Spot vs Binance USDⓈ-M Futures：现货与期货的 mid 短暂偏离 1-5 bps 是常态；
- Co-located 在 AWS Tokyo (ap-northeast-1) 的 EC2 节点到 Binance Matching Engine 延迟 < 1ms，可以 *秒级* 捕获价差。

**跨所 (Cross-venue) Latency Arb**：
- Binance → OKX → Bybit 三站 *TCP* 延迟 50-200ms；
- 通过 *Ultra-Low Latency (ULL) Microwave* 链路 (Hudson Fiber, Anova) 可压缩到 5-10ms，但成本 \$1M+/月；
- 大型 prop 仍靠 *FPGA* 实现微秒级 order generation (e.g., Jane Street 的 Tokyo cluster) [[Co-location Wikipedia](https://en.wikipedia.org/wiki/Co-location)]。

### 6.5 Cross-Venue Rebate Farming

每家交易所都提供 *maker rebate* (返佣)：
- Binance Spot VIP9: 0.012% maker rebate
- OKX VIP9: 0.015% maker rebate
- Bybit VIP5: 0.025% maker rebate
- Coinbase Advanced: 0.05% maker rebate (USDC 交易对)

**策略**：在多家交易所 *同时挂单* 赚取多个 rebate，需要 *cross-venue inventory 中性*：
- 当 Binance 成交 buy → 立即在 OKX 卖等量，保持 net inventory ≈ 0；
- 月回报 = $\sum_i \text{volume}_i \times \text{rebate}_i$；
- 风险：*latency disbalance* 导致单边成交 → inventory spike；2022-06 Wintermute 因 *multisig bug* 损失 \$160M 即是 *operational* 风险。

### 6.6 Funding Rate 套利详解

Funding rate 套利 4 种主流变体：

| 策略 | 多头 | 空头 | 净收益 | 风险 |
|---|---|---|---|---|
| **Long Spot + Short Perp** | Buy spot | Short perp | funding - basis | perp 爆仓, basis 收窄 |
| **Long Perp + Short Spot** | Long perp | (借币) Short spot | -(funding + 借币费) | borrow 费 > funding |
| **Cash-and-Carry** | Buy spot | Sell quarterly future | basis | basis 收窄, rollover 风险 |
| **Reverse Cash-and-Carry** | (借币) Short spot | Buy quarterly | -basis + borrow | perp 升水时 trigger |

实操 (BTC funding=10% APR, basis=12% APR)：
- Long spot BTC \$1M + short perp BTC \$1M (notional)
- 月收入 = funding 0.83% + basis 摊销 0.5% = 1.3% (15% APR)
- 风险：当 perp 暴跌 > 50%, perp margin call, *auto-deleveraging* 强制平仓；最坏 2020-03-12 当日 BTC 跌 50% 触发大量 liquidation。

### 6.7 TWAP / VWAP / POV 执行算法

**TWAP (Time-Weighted Average Price)**：把大单均分到时间窗 $[0, T]$：
- $\text{order}_t = Q / T$ 每秒等量
- 适合 *隐私* 优先 + 短窗口

**VWAP (Volume-Weighted Average Price)**：按历史 volume curve 分配 (e.g., U-shape)：
- 适合长窗口，与市场 participation 同步 [[VWAP Wikipedia](https://en.wikipedia.org/wiki/Volume-weighted_average_price)]

**POV (Percentage of Volume)**：实时维持 $\alpha$% 的 market share：
- 适合 *adaptive*，vol 放大时自动减速

**Implementation Shortfall (Almgren-Chriss 2000)**：最小化 *expected cost = 临时冲击 + 永久冲击 + 风险* 的 objective 优化：
- $\text{cost} = \eta \cdot v + \frac{1}{2} \gamma v^2 + \lambda \cdot \sigma \cdot \sqrt{T}$
- 给出 *optimal trajectory* $x^*(t) = \sinh(\kappa(T-t)) / \sinh(\kappa T)$
- 这是 *execution* 的"教科书解"，但需要实时 $\sigma$ 估计。

### 小结 (六)

激进策略 (taker / aggressive) 与做市策略 (maker / passive) 是订单簿策略的两个互补维度。激进的 alpha 来源主要是 (a) 速度 (b) 跨所错位 (c) 套利窗口。技术壁垒包括 *co-location*、*FPGA*、*微波 / 激光* 链路。Funding Rate 套利是 *无方向风险* 的最大 alpha 池之一，2024 年 BTC funding 长期高位 (5-15% APR) 创造了大量 delta-neutral capital。

## 七、盘口异常与操纵检测

### 7.1 Spoofing 检测

**Spoofing** (伪装挂单) 定义：在大单 (但远端) 价格挂出 *不会执行* 的 limit order，制造虚假需求/供给；当价格向 spoof 方向移动后立即 cancel 整个 spoof。

**检测方法 (Bouchaud 2017, "The Fatter the Tail, the Slower the Convergence")**：
1. **Order-to-Trade Ratio (OTR) Filter**：当某地址 OTR > 100 且 cancel < 1s 即撤，标记为可疑；
2. **Queue Position Anomaly**：远端 (level > 10) 突然出现 size > 5x 中位 depth 的挂单，且 < 1s 内 cancel → 95%+ 概率是 spoof；
3. **Post-trade Cancel**：当 trade 触发 *taker 突破* 1 个 level 后，spoofer 立即撤走其所有 spoof 订单；用 *event study* 找 *post-event cancel > 80% of total depth* 的 pattern。

> **法律参考**：美国 Dodd-Frank Act §747 与 CFTC Reg 180.1 把 spoofing 列为 *manipulative or deceptive device*；2020-10-01 JPMorgan 同意支付 \$920M 因 *spoofing* 在金属 / 期货市场的指控 (CFTC Press Release 8704-23) [[CFTC PR 8704-23](https://www.cftc.gov/PressRoom/PressReleases/8704-23)]。

### 7.2 Layering / Quote Stuffing

**Layering** 是 spoofing 的多档版本：在多个 price levels 同时挂大单形成 *假墙*，吸引散户 *front-run*；一旦价格朝目标方向移动，spoofer 立即 *cancel* 所有远端层。

**Quote Stuffing** (操纵者在毫秒内 insert + cancel 数千订单，制造"流动性"假象并消耗 HFT 算力)：
- 2010 NASDAQ 多次发生，导致 *latency spike*；
- 加密盘口更易实施：BTC 现货 *insert-cancel* 频率中位 5-10 笔/秒，事件期内可冲到 1000+ 笔/秒；
- 检测：监测 *insert-then-cancel within 50ms* 的 *mass events* (>100 笔/100ms) [[Quote Stuffing Wikipedia](https://en.wikipedia.org/wiki/Quote_stuffing)]。

### 7.3 Wash Trading (Bitwise 报告)

**Wash Trade** 定义：同控制下的多个账户 *互为对手方*，制造虚假成交量 (美国 7 USC §13)。

**Bitwise 报告 (2019-03, "Manipulation, Money Laundering, and Wash Trading in Crypto Markets")** 估算：
- 95% 的 *reported* BTC 现货交易量是 *fake* 或 *non-economic*；
- 全市场 wash trade 月额 > \$1B；
- 主要发生在 *Binance* (low fee), *Bitfinex*, *OKEx*。

**检测方法**：
1. **Self-match detection**：同一 *user_id* 在 bid/ask 两边同时成交，立即 reverse；
2. **Address clustering**：链上分析 (e.g., Chainalysis, Nansen) 找出 *controlled-by-same-entity* 的 addresses；
3. **Statistical detection**：用 Hawkes process 估计 *non-random order arrival* pattern。

> Bitwise 报告引用 87% 的 Spot Volume 估算为 wash trade；2021 年 The Block Research 修正估算为 60-70% (剔除洗盘)，2023 年 The Block 把全市场 *真实* 现货 volume 估算在 \$5-10B/日 (vs reported \$50-100B) [[The Block Research](https://www.theblockresearch.com/)]。

### 7.4 Pump & Dump (拉盘砸盘)

Pump-and-dump 是 *fraud* 的标准模型：
1. **Pump phase**：组织者在 Telegram / Discord 群组宣布 *pump time*，成员 *集中买入* 某小币，price 10x-100x；
2. **Dump phase**：组织者 *先 dump*，成员 *慢* 一拍接盘。

**检测**：
- 1-min price change > 30% + volume > 5x 中位 → *pump candidate*；
- 链上 (on-chain) 监测 *token contract* 的 *new holder* 数量急剧增加 (e.g., > 200/分钟)；
- 2021-05 *Save the Kids* (Kris Jenner coin) 等案例；SEC 多次起诉 *Celebrity-backed coins* (e.g., 2022 Kim Kardashian / EthereumMax 罚款 \$1.26M)。

### 7.5 闪崩案例

**2017-03 ETH Flash Crash**：2017-06-21 GDAX (Coinbase Pro 前身) ETH 闪崩至 \$0.10 后反弹；触发 *stop-loss chain* + *maker cancel all*；详细复盘见 Coinbase 官方报告。

**2021-05-19 BTC Flash Crash**：BTC 在 2021-05-19 亚洲早盘从 \$43K 跌至 \$30K (-30%)，主因：
- Elon Musk 关于 Tesla *暂停接受 BTC 支付* 的 tweet 引发恐慌；
- 大量 *leveraged long* 爆仓 + *stop-loss cascade*；
- 交易所系统过载，Binance、OKX 部分 *fails to deliver* margin call。

**2024-08-05 Global Crypto Flash Crash**：BTC 24h 跌 15% 至 \$49K，ETH 跌 22%，主要因 *carry trade unwind* (日元 carry) + 弱宏观 + 周末低流动性；Fed / BIS 都关注此事 [[Federal Reserve FEDS Notes 2024](https://www.federalreserve.gov/)]。

### 7.6 监管：SEC / CFTC / MiCA

**美国**：
- **SEC** 监管 *securities* 性质的代币 (e.g., XRP, SOL, ADA 历史争议)；
- **CFTC** 监管 *commodity* 性质的 BTC / ETH 期货 / perpetuals；
- 2024-01 SEC 批准首批 11 只 *Bitcoin Spot ETF* (BlackRock IBIT, Fidelity FBTC 等)；2024-05 批准 9 只 *ETH Spot ETF* (未含 staking)。

**欧盟 MiCA (Markets in Crypto-Assets)**：
- 2023-06 published in OJEU，2024-06-30 全面生效；
- 强制 *CASP* (Crypto-Asset Service Provider) 牌照，MiCA 涵盖 *asset-referenced tokens* (ART) 和 *e-money tokens* (EMT)；
- 限制 *non-EU stablecoin* 在单一交易额 €200M 以内；
- 影响：2024-08 Binance 退出欧洲部分市场 (Cyprus, Netherlands, Germany) 重组为 MiCA 合规实体 [[ESMA MiCA](https://www.esma.europa.eu/)]。

**CFTC 加密执法 (2022-2024)**：
- 2022-12 Ooki DAO 起诉 (CFTC PR 8586-23)：CFTC 起诉 *Ooki DAO* (后成为 *未注册实体*)；
- 2023-09 Binance.US 与 SEC 诉讼 (Securities 性质争议)；
- 2023-10 FTX 创始人 Sam Bankman-Fried 7 项重罪成立。

### 小结 (七)

订单簿操纵检测是 *合规做市* 的核心能力。Spoofing / Layering / Quote Stuffing 在加密市场比传统股票市场 *更易实施* (因低监管成本 + 跨境套利空间)。Bitwise 报告与 The Block Research 的 *真实 volume 估算* 是市场结构研究的关键参考。监管层面 SEC / CFTC / MiCA 三大法域在 2023-2024 加速落地，监管差异是 *流动性迁移* 与 *策略 ROI 衰减* 的重要外生变量。

## 八、订单簿机器学习 (ML for Order Book)

### 8.1 DeepLOB (Zhang 2019, arXiv 1807.02328)

Zhang, Zohren, Roberts (Oxford-Man Institute) 2019 在 arXiv 1807.02328 提出 DeepLOB：CNN + LSTM 混合架构，输入 100 笔 L2 深度 + 10 笔 trade，输出 3-class softmax (涨/平/跌)：

- **Architecture**：3 个 Conv1D 块 (kernel 1, 2, 4) → 拼接 → 2 层 LSTM (64 hidden) → 全连接；
- **Dataset**：NASDAQ ITCH, 2014-2018, 5 stocks (AMZN, GOOG, INTC, AAPL, MSFT)；
- **Accuracy**：mid-price 50-tick 预测准确率 67-80% (vs baseline 33% / 50% random)；
- **F1 (macro)**：0.42-0.62；
- **关键 feature**：*top-10 levels 100 ms 滚动窗口* + *volume imbalance*。

DeepLOB 的设计哲学是 *特征工程 + 深度学习* 双管齐下；其 input representation (`(price, qty) × 2 sides × 10 levels × 100 timesteps`) 成为后续 LOB-Net 论文的 de-facto 标准 [[DeepLOB arXiv 1807.02328](https://arxiv.org/abs/1807.02328)]。

### 8.2 TransformerLOB

TransformerLOB (2021-2022, 多篇) 把 DeepLOB 的 LSTM 替换为 Transformer encoder：
- 优势：长程依赖（100+ ticks）；
- 劣势：纯 self-attention 在 100ms 时间尺度上不一定优于 LSTM；
- 实测：在 BTC-USDT 1s 预测上准确率 60-65%，比 DeepLOB 提升 3-5%。

### 8.3 GNN 建模盘口

Graph Neural Network (GNN) 把订单簿看作 graph：每个 price level 是 node，边代表 *order flow between levels*。代表论文：
- **LOB-GNN (Tsai et al. 2022)**：spatial + temporal 双图卷积；
- **DL-GNN (2023)**：*Dynamic LOB Graph*，节点是 (price, qty)，边是 trade 路径。

在 BTC 现货回测中，GNN 比 DeepLOB 在 *跨交易所 / 跨币种* 迁移上更稳健。

### 8.4 强化学习做市

**J.P. Morgan 2019 (Buehler et al., arXiv 1809.10039 "Deep Hedging")**：
- 用 actor-critic (PPO) + LSTM policy 对冲 *delta-gamma* 暴露；
- 在 S&P 500 模拟 + 2017 BTC 实际数据上，deep hedging 比 *delta-only hedge* 提升 25% risk-adjusted return。

**BNP Paribas 2023 (arXiv 2304.10248, "Reinforcement Learning for Market Making")**：
- 模拟 5 个主要 asset class (FX, crypto, equity, rates, commodity)；
- 关键设计：(a) *inventory penalty* 嵌入 reward (b) *adverse selection cost* 通过 mark-out 估计 (c) *soft actor-critic* + 3-layer LSTM；
- 在 crypto 模拟中，RL MM 的 PnL 提升 30% (相对 Avellaneda-Stoikov)，Sharpe 提升 0.5-0.8 [[BNP Paribas 2023](https://arxiv.org/abs/2304.10248)]。

**其他代表工作**：
- **DeepDirectRL (Lim et al. 2019)**：用 policy gradient 直接优化 PnL；
- **Spooner et al. 2018 (arXiv 1802.04177)**：做市 RL benchmark；
- **Sadighian (2019) "Deep Reinforcement Learning in Cryptocurrency Market Making"**：BTC 现货 RL MM 实验，提示 PPO 收敛慢且易过拟合。

### 8.5 合成数据训练

由于真实盘口数据 *带标签成本高* (只有事后才能定 mid)，研究者常用：
1. **Stylized Fact Simulator (Cartea-Jaimungal-Penalva 2015)**：基于 Hawkes process 的合成 LOB，参数 $(\lambda_\text{limit}, \lambda_\text{cancel}, \lambda_\text{market})$ 校准真实数据；
2. **Multi-Agent Simulation (MAS)**：用 *zero-intelligence* + *informed-trader* 模拟产生 "fact-rich" 训练数据；
3. **Diffusion Model 生成 (2024)**：用 DDPM 在历史 LOB snapshot 上 *conditional generation*，实现 "what-if" 训练。

> **陷阱**：合成数据可能丢失 *adverse selection* 的 microstructure tail——这恰恰是 live trading 的最大 PnL 来源。**永远**要把模型在真实 out-of-sample 数据上做最终评估。

### 小结 (八)

ML 在订单簿策略的渗透率从 2018 (DeepLOB) 持续加速；2023-2024 出现 *RL 做市商* (BNP Paribas) 的工业级论文。ML 的核心价值是 (a) 自动化特征工程 (b) 实时 decision (c) 自适应 regime change。**关键陷阱**：(1) 数据有 *非平稳性*，不能简单 i.i.d. 训练；(2) *adverse selection* 难以建模 → RL 需用 *mark-out reward*；(3) *合成数据* 不能替代真实数据。

### 9.2 特征存储：kdb+ / QuestDB / ArcticDB / DuckDB

特征 (feature) 的存储与查询是订单簿策略的 *基础设施*。

| 工具 | 类型 | 优势 | 劣势 |
|---|---|---|---|

- 冷数据 (历史 tick, 1B+ rows)：Parquet on S3 + ArcticDB；
- 极致性能需求：kdb+ on premise (NYC LD4 / Tokyo TY3 co-located) [[ArcticDB docs](https://docs.arcticdb.io/)] [[QuestDB](https://questdb.io/)] [[DuckDB](https://www.duckdb.org/)]。

### 9.3 回测陷阱：L3 幸存者偏差

回测订单簿策略最常见的 *幸存者偏差 (survivorship bias)* 来自：

1. **L3 序列错位**：如果用 *raw 增量* (没有正确处理 `(lastUpdateId, prevUpdateId)` 重同步)，在丢包段会 *假设盘口不变*，导致 backtest PnL 严重虚高。
2. **数据 vendor 切片偏差**：Kaiko、Tardis.dev、Databento 都提供 *historical* L2/L3，但 *symbol list* 仅包含 *当前* 在交易所的 symbol (剔除 *delisted* 的)；这意味着 *delisted* 的 *flash crash* 事件 *消失* 在数据里 [[Tardis.dev docs](https://docs.tardis.dev/)]。
3. **Maker 优先级偏差**：在 backtest 中默认做市商 *总在 top*，但真实场景下 *queue position* 决定成交——这是 *fictitious alpha* 的主要来源。
4. **Slippage 估计偏低**：Tardis / Databento 提供 *top-of-book* 历史，但 *next-level fill* 的 actual price 需要 *l2-update* 严格回放。
5. **Regime Change**：用 2020-2021 牛市数据训练 GLFT，在 2022 熊市会严重 *over-spread*。

**回测标准操作**：
- 至少 3 个 regime (bull / bear / sideways) 验证；
- 跑 *paper trading* 至少 30 天；
- *out-of-sample* 留 6 个月；
- *Mark-out PnL* 必须正向 (5s, 30s, 5min 三档)。

### 9.4 滑点：Almgren-Chriss 2000 Square-Root

Almgren-Chriss 2000 ("Optimal Execution of Portfolio Transactions", J. Risk 3(2)) 是 *execution cost* 的标杆模型：

$$
\text{impact}(Q) = \alpha \cdot \sigma \cdot \sqrt{Q / V_{\text{day}}}
$$

其中 $Q$ 是 order size，$V_{\text{day}}$ 是 daily volume，$\alpha$ 是 *tier-dependent* 因子。实测 (BTC 现货, Binance 2024)：
- $Q = \$1M$ → $\text{impact} \approx 2$ bps；
- $Q = \$10M$ → $\text{impact} \approx 8$ bps；
- $Q = \$50M$ → $\text{impact} \approx 25$ bps；
- $Q = \$200M$ → $\text{impact} \approx 80$ bps。

**Twist**: Almgren-Chriss 假设 *永久冲击*，但加密市场 *非永久* 冲击比例高 (>50%)——可以加 *recovery factor* $\beta$：

$$
\text{impact}(Q, t) = \alpha \sigma \sqrt{Q / V_{\text{day}}} \cdot e^{-\beta t}
$$

> **实战**：在 *执行* 大宗交易时，*split* 为 10-20 个 child order 跨 1-2 小时执行，*expected impact* 减少 30-50%。

## 十、跨资产联动与期权做市

### 10.1 现货 / 永续 / 季度期货三角套利

加密市场的 *cross-asset* 三角套利比传统股票市场丰富得多——单一标的 BTC 至少有 3 个 trading venue：

1. **Spot**：Binance Spot BTC-USDT
2. **Perpetual Swap**：Binance USDⓈ-M BTC Perpetual (每 8h funding)
3. **Quarterly Future**：CME BTC Future (CME Group 2024 起; Binance / OKX 季度合约)

三角套利的核心方程：
$$
\text{arb}_{t} = P^{spot}_t \cdot (1 + r_{\text{free}}) - P^{perp}_t \cdot e^{-r_{\text{funding}} \cdot T} - P^{quarterly}}_t
$$

实战 (2024-01 BTC ETF 批准前 1 周)：
- Spot \$42,000, Perp \$42,200 (8h funding = 0.04%, 8h)
- Quarterly (Mar 2024): \$43,500
- 套利空间：Long Spot + Short Perp, 锁定期现价差 (basis) 3.6% / 季，扣除 funding 0.3% / 月 ≈ 0.7% / 月净收益。
- 风险：perp > spot 价差 *突然消失* (regime change)，或者交易所 *short margin* 利率突然上升。

### 10.2 期权订单簿与做市

Deribit 是加密期权的 *核心场所* (2024 年市占率 80%+)。期权做市比 *现货 / 永续* 复杂 5-10 倍：

- **Two-Sided Quotes on 3-D Surface**：要同时在 *strike × expiry × side* 三维做市；
- **Greeks 中性**：做市商必须保持 (delta, gamma, vega, theta) 在预算内；
- **Vol Surface 套利**：在 vol surface 偏离 *no-arbitrage condition* (Put-Call Parity, Calendar Spread Arbitrage, Butterfly Arbitrage) 时套利。

Deribit 订单簿结构：
- 单一 option (e.g., BTC-2024-12-27-100000-C) 有 100 strike × 1 expiry，4-5 price levels 深度；
- WebSocket API 推送 100k+ msg/s 整体 (所有 option 合计)；
- 主流机构 (Wintermute, Genesis, Cumberland/DRW, Galaxy Digital) 是 maker [[Deribit docs](https://docs.deribit.com/)]。

**Vol surface 套利 (PCA 框架)**：
- 主成分分析 (PCA) 提取 vol surface 三个主成分 (level, slope, convexity)；
- 当某个主成分偏离其 1σ 区间 → 套利；
- 收益：5-10% APR，sharpe 1.5+；
- 风险：gamma blow-up 当 underlying 跳空 (加密 options 在 2024-08-05 flash crash 时 IV 跳到 90% vol)。

### 10.3 DeFi AMM 与 CLOB 联动

Uniswap v3、Curve、PancakeSwap 等 AMM 在 *长尾币种* 上仍是主要交易场所；机构正在 *CLOB ↔ AMM* 之间套利：
- 当 AMM 报价偏离 CLOB 1-5% 时，用 *atomic arbitrage* 套利 (single tx in 同一区块)；
- 主要 DEX 三明治 (sandwich) 套利 / JIT (just-in-time) liquidity 提供，2022-2023 MEV 盛行。

*订单簿策略* 视角：把 AMM 看作 "无穷多档 + 滑点连续" 的 limit order book，可以借鉴 CLOB 策略到 AMM 池子 (LP provision as a quasi-market-making) [[Uniswap Wikipedia](https://en.wikipedia.org/wiki/Uniswap)]。

### 小结 (十)

跨资产 (现货 / 永续 / 季度) 与期权订单簿是 *机构级* 做市商的 *必然战场*。Deribit 在期权上占据主导，vol surface 套利 sharpe 1.5+ 但需 gamma 管理；DeFi AMM 是 *长尾币种* 的必要补充，*MEV-aware* 的跨 CLOB-AMM 套利是新兴方向。加密市场的 *multi-asset* 复杂度，是 *传统股票对冲基金* 的 3-5 倍。

### 11.1 OBI 计算 (Python)

```python
import numpy as np
import pandas as pd
from typing import Tuple

class OrderBook:
    """简化的 L2 订单簿 (Binance Spot 风格)"""
    def __init__(self, depth: int = 20):
        self.depth = depth
        self.bids: dict[float, float] = {}  # price -> qty
        self.asks: dict[float, float] = {}
    
    def update(self, price: float, qty: float, side: str):
        if side == 'bid':
            if qty == 0: self.bids.pop(price, None)
            else: self.bids[price] = qty
        else:
            if qty == 0: self.asks.pop(price, None)
            else: self.asks[price] = qty

    def top_n_qty(self, side: str, n: int) -> float:
        if side == 'bid':
            prices = sorted(self.bids.keys(), reverse=True)[:n]
        else:
            prices = sorted(self.asks.keys())[:n]
        return sum(self.bids.get(p, 0) for p in prices) if side == 'bid' \
            else sum(self.asks.get(p, 0) for p in prices)

    def obi(self, levels: int = 1) -> float:
        q_b = self.top_n_qty('bid', levels)
        q_a = self.top_n_qty('ask', levels)
        return (q_b - q_a) / (q_b + q_a) if (q_b + q_a) > 0 else 0.0

    def microprice(self) -> float:
        """Stoikov microprice 公式"""
        q_b = self.bids.get(max(self.bids), 0)
        q_a = self.asks.get(min(self.asks), 0)
        p_b = max(self.bids)
        p_a = min(self.asks)
        return (p_b * q_a + p_a * q_b) / (q_b + q_a)
```

### 11.2 GLFT 做市报价 (Python, 简化)

```python
import math

def glft_quote(sigma: float, gamma: float, kappa: float, T: float, 
               q: float, mid: float, side: str) -> float:
    """
    GLFT 2013 简化报价函数
    sigma: 年化波动率 (例如 BTC = 0.6)
    gamma: 风险厌恶 (例如 0.5)
    kappa: 订单簿弹性参数 (例如 1.0)
    T: horizon (年, 例如 0.001 = 几小时)
    q: inventory
    side: 'bid' or 'ask'
    """
    # 简化的 reservation price 偏置
    reservation = mid - q * gamma * sigma**2 * T / 2.0
    # 基础 spread (half)
    half_spread = (1.0 / kappa) * math.log(1.0 + gamma * kappa * sigma**2 * T / 2.0)
    
    if side == 'bid':
        return reservation - half_spread
    else:
        return reservation + half_spread

# Example: BTC 现货
mid = 70000
sigma = 0.6  # 60% 年化
gamma = 0.5
kappa = 1.0
T = 0.001    # 约 1 小时

for q in [-3, -1, 0, 1, 3]:
    bid = glft_quote(sigma, gamma, kappa, T, q, mid, 'bid')
    ask = glft_quote(sigma, gamma, kappa, T, q, mid, 'ask')
    print(f'q={q:+d}: bid={bid:.2f}, ask={ask:.2f}, spread={ask-bid:.2f}')
```

```python
    start_http_server(8000)  # Prometheus 抓取端口
    while True:
        # main loop, 调用 monitor()
        time.sleep(0.1)
```

## 十二、订单簿策略的风险管理

### 12.1 三层风险模型

订单簿策略的风险管理分为三层：

1. **Position Risk (仓位风险)**：单一 symbol / 单一 venue 的 inventory 暴露；
2. **Cross-Venue Risk (跨场所风险)**：多个 venue 的 net exposure (e.g., Binance 长 + OKX 短 = net 0，理论上无方向风险)；
3. **Systemic Risk (系统性风险)**：交易所破产、监管禁令、链上黑客事件。

**Position Limit (仓位上限)**：
- BTC 现货单 symbol: $q_{\max} = 2\text{-}5$ BTC (中规模做市商)；
- ETH 现货: $q_{\max} = 50\text{-}200$ ETH；
- 小币种 (top-50 之外): $q_{\max} = 0.5\text{-}1\%$ of 24h volume；
- 跨所 net exposure: $\sum_i q_i \cdot P_i \leq \text{var\_budget}$。

### 12.2 VaR 与 Expected Shortfall

**Historical VaR (1-day, 99%)**：
$$
\text{VaR}_{0.99} = -\text{quantile}_{0.01}(R_t)
$$
其中 $R_t$ 是 PnL 序列。BTC 现货做市 1-day VaR 通常 0.5-1.5% (sharp risk management)。

**Expected Shortfall (CVaR, 99%)**：
$$
\text{ES}_{0.99} = -E[R_t | R_t < \text{VaR}_{0.99}]
$$
ES 捕捉 *尾部事件* 的平均损失，比 VaR 严格；机构普遍用 ES 作为 *capital 充足率* 指标 (Basel III/IV 框架)。

### 12.3 Drawdown 控制

- **Max Drawdown Limit**：5% 触发减仓，10% 触发全面停止做市；
- **Recovery Time**：从 drawdown 恢复超过 30 天 → 策略 review；
- **Consecutive Loss**：连续 5 日亏损 > 2σ → 暂停策略。

加密市场常见 drawdown 案例 (BTC 现货做市 2022 熊市)：12 月至次月 *rolling 1-month max DD* 8-15%，远高于股票市场 2-3%。

### 12.4 Operational Risk

Operation 是 *op risk* 的最大来源：
1. **API 故障**：Binance 2021-11-03 现货撮合短暂宕机 (30 分钟)；
2. **WAF / Network Issue**：2022-11-09 FTX 系统性瘫痪 (破产前)；
3. **Smart Contract Hack**：Wintermute 2022-09 被攻击损失 \$160M (DeFi 私钥管理)；
4. **Counterparty Solvency**：FTX / Alameda 2022-11 破产；BlockFi 2022-11 破产；
5. **Regulatory Ban**：2021-09 中国大陆全面禁止 crypto 交易 (一夜清空流动性)；
6. **Key Person Risk**：单一策略 owner 离职 / 意外 → 策略失效。

**防御手段**：
- 多 venue 分布式执行 (e.g., Binance 50% + OKX 30% + Bybit 20%)；
- 实时 PnL 风险 dashboard (Grafana)；
- PGP / HSM 私钥管理；
- Legal entity 分隔 (proprietary trading vs. client funds)；
- *kill switch* 一键清仓机制 (合规要求)。

### 12.5 Tail Risk (黑天鹅)

加密市场的 *tail event* 比传统股票市场 *3-5x* 频繁：
- 2020-03-12 BTC 24h 跌 50% (COVID panic + margin call cascade)；
- 2021-05-19 BTC 24h 跌 30% (Tesla 反转 + margin call)；
- 2022-11-09 FTX 崩盘，BTC 24h 跌 15%；
- 2024-08-05 BTC 24h 跌 15% (carry trade unwind)。

*Tail risk* 防御：
- 长期 *vega positive* 期权仓位 (买 OTM put)；
- 跨资产对冲 (BTC 长 + S&P 500 短)；
- 现金 / 稳定币 reserve > 30% NAV；
- 极端减仓规则：1-day loss > 5% → *flatten all*。

### 小结 (十二)

风险管理是订单簿策略的 *生存基础*。三层风险模型 (Position / Cross-venue / Systemic) + VaR / ES / Drawdown 三种度量 + Operational risk 防御 = 完整 *风控体系*。加密市场 tail event 频繁 (3-5x 传统股票)，做市商必须时刻为 *5% 日内跌* 准备；Alameda 2022 的倒闭再次提醒 *concentration* + *related-party* 风险的毁灭性。

## 十三、24/7 运营与人员配置

### 13.1 加密市场的 24/7 性质

加密市场 *永不关闭*，这是与股票市场最大的运营差异：

- **跨时区流动**：Asia (Tokyo 09:00-15:00 JST) → EU (London 08:00-16:00 GMT) → US (NY 09:30-16:00 EST) 三班倒；
- **周末 / 假期低流动性**：周末交易量是工作日的 30-50%，vol 反而上升 1.5-2x；
- **节假日事件**：Christmas (12-25) / Chinese New Year 等出现 *gap risk*；
- **突发事件**：tweet / regulatory 公告 / 黑客事件 → 5 分钟内可能 -10%。

### 13.2 团队配置 (中型机构)

- **Quant Researcher (3-5 人)**：策略开发、回测、信号工程；
- **Quant Developer (3-5 人)**：生产代码、Rust/C++/Python 维护、ML ops；
- **Trader (2-3 人)**：live trading 监控、人工干预 (突发事件)；
- **Risk Manager (1-2 人)**：风控规则、限额、监管报告；
- **Ops (1-2 人)**：基础设施、监控、告警 (24/7 on-call)；
- **Compliance / Legal (1-2 人)**：KYC/AML、MiCA/FinCEN 报告。

### 13.3 工作时间覆盖

- **Asia 班** (HK / SG 09:00-18:00)：覆盖 China / Japan / Korea 时段
- **EU 班** (LDN 08:00-17:00)：覆盖欧洲 + 中东
- **US 班** (NY 09:00-18:00)：覆盖美国 + LATAM
- **夜间 / 周末**：on-call 轮值，*kill switch* 待命

**典型事件** (2022-11-09 FTX 崩盘) 团队反应：NY 班 trader 在 19:00 EST (凌晨 Asia) 收到 telegram 告警 → 30 分钟内 *flatten all*；平均 *time-to-react* 10-20 分钟。

### 13.4 自动化与人工干预的边界

*机器做* (95% 时间)：
- 日常 quoting, cancel-replace, inventory skew
- 风险监控、告警分发
- 数据采集、特征计算

*人工做* (5% 时间)：
- 极端事件 (regulatory, hack, exchange bankruptcy) → *kill switch* 决策；
- 策略参数调整 (regime change 检测)；
- 跨所 routing 优化 (incentive, fee schedule 变化)；
- 模型 review (overfitting detection)。

### 小结 (十三)

24/7 运营是加密做市的 *不可回避* 成本。机构团队至少 10-15 人 / 班次覆盖；自动化覆盖 95% 的常规工作，但 *极端事件* 必须有人工干预 (kill switch / 跨所迁移)。*时间覆盖* 不到位的小团队，是 *tail risk* 的最大受害者。

## 十四、未来研究方向与开放问题

### 14.1 L3 数据开放与监管沙盒

当前加密交易所的 L3 数据被 *封闭*，但监管沙盒 (如英国 FCA, 香港 SFC, 阿联酋 VARA) 正在推动 *transparency*。若 L3 数据在 2025-2027 年间逐步开放，*order-flow disaggregation* 类的学术研究将迎来 *黄金时代*，可能产生新的 *foundational 论文* (类比 2010 后的 L2 学术浪潮)。

### 14.2 RL + LLM 双层决策

2024 年开始出现的 *LLM-augmented RL* (e.g., FinGPT, BloombergGPT) 探索把 *宏观叙事* (e.g., Fed minutes, SEC filings) 与 *微观 LOB 数据* 联合建模。预计 2025-2027 年间会出现 *narrative-driven 做市商*——LLM 读取 *tweet / news*，RL 调整 *quote skew*。

### 14.3 跨 CeFi-DeFi 一致性

随着 *DEX aggregator* (1inch, Matcha, CowSwap) 成熟，*CeFi-DeFi 套利* 将更高效。订单簿策略需要把 *on-chain order book* (e.g., 0x, Loopring, dYdX v4) 与 *CLOB* 看作 *unified state* 来优化执行。

### 14.4 量子计算对 ECDSA / Ed25519 的威胁

BTC / ETH 使用 ECDSA secp256k1，量子计算 (Shor 算法) 成熟后将威胁 *私钥*。预计 2025-2030 出现 *post-quantum cryptography* 升级 (NIST 2024-08 公布 ML-DSA, SLH-DSA 标准)，做市商需要 *legacy vs post-quantum* 的过渡期管理。

### 14.5 AI 生成订单簿操纵

*Generative Adversarial Networks* (GAN) / *Diffusion Models* 可用于 (a) *spoof 制造* 与 (b) *检测*。这是一把 *双刃剑*：监管需要 *AI-based surveillance* 配套，*检测者与造假者* 的军备竞赛将是 2025+ 的核心议题。

### 14.6 Tokenization 订单簿

现实世界资产 (RWA, Real World Asset) 正在上链——美债、股票、地产、私募基金份额。预计 2025-2027 年，*RWA 订单簿* 将与 crypto 现货订单簿 *融合*，做市商需要 *multi-asset class* 框架 (e.g., BlackRock BUIDL, Franklin Templeton FOBXX, Ondo Finance USDY)。

### 小结 (十四)

订单簿策略是 *最经久不衰* 的 quant 话题，从 1985 (Kyle) 到 2024 (RL MM, BNP Paribas) 持续演化。未来的突破可能在 (a) L3 数据开放 (b) LLM + RL 融合 (c) CeFi-DeFi 一致性 (d) 量子 / 后量子 (e) AI 监管 (f) RWA tokenization。**做市商必须保持技术敏感度，把 *alpha hunt* 与 *infra investment* 持续结合**。

## 十五、结语：订单簿策略的本质

订单簿是金融市场的 *物理底层*，而订单簿策略是量化交易的 *艺术与科学*。

**本质一**：alpha 来自 *微观结构*，不是 *macro / fundamental*。懂 LOB 的 microstructure，胜率比不懂的高 5-10x。

**本质二**：速度 = 钱。在同质化的策略中，*延迟差 1ms = 1% PnL*。Co-location 与 FPGA 不是奢侈品，是 *生存工具*。

**本质三**：风险 > 收益。做市商的 *长期生存* 取决于 *risk management*，不是 *alpha generation*。Alameda / FTX 的崩溃就是 *risk mismanagement* 的代价。

**本质四**：监管 = alpha。SEC / CFTC / MiCA 改变 *流动性地图*，*合规做市商* 比 *灰色做市商* 享有 5-10x 容量。

**本质五**：人是关键。模型、infra、策略都是 *死的*；人才是 *活的*。一个 *核心 quant* 的离开可以让 *百亿* 资产瞬间崩溃。

> **最后一句**：订单簿策略不是 *alpha hunt*，而是 *signal × speed × risk × ops × people* 的系统工程。愿每个读到此处的 researcher / trader / engineer 都能在 *微观结构* 中找到自己的 alpha。

## A.1 L2/L3 订单流 WebSocket 拉取（Binance / OKX）

```python
"""
实时 L2 (top 20 深度) 订单流拉取
- 异步 websockets，避免阻塞
- 落盘 to Parquet (DuckDB 可查)
"""
import asyncio
import json
import websockets
import pandas as pd
from datetime import datetime


async def stream_binance_l2(symbol: str = "btcusdt",
                            out_path: str = "l2.parquet",
                            max_msgs: int = 10000) -> None:
    url = f"wss://stream.binance.com:9443/ws/{symbol}@depth20@100ms"
    rows = []
    async with websockets.connect(url, ping_interval=20) as ws:
        for i in range(max_msgs):
            msg = await ws.recv()
            data = json.loads(msg)
            ts = datetime.utcfromtimestamp(data["E"] / 1000)
            for px, qty in data["bids"][:5]:
                rows.append({"ts": ts, "side": "bid", "level": 1, "px": float(px), "qty": float(qty)})
            for px, qty in data["asks"][:5]:
                rows.append({"ts": ts, "side": "ask", "level": 1, "px": float(px), "qty": float(qty)})
            if (i + 1) % 1000 == 0:
                pd.DataFrame(rows).to_parquet(out_path)
                print(f"[{i+1}] saved {len(rows)} rows")
    pd.DataFrame(rows).to_parquet(out_path)
```

## A.2 OBI / Microprice / OFI 指标计算

```python
"""
三大基础信号
- OBI (Order Book Imbalance): (bid_qty - ask_qty) / (bid_qty + ask_qty)，范围 [-1, 1]
- Microprice: 加权中间价
  micro = (bid_px * ask_qty + ask_px * bid_qty) / (bid_qty + ask_qty)
- OFI (Order Flow Imbalance): Cont 2021 论文
  ΔV_b = max(0, b_n - b_{n-1}) - max(0, b_{n-1} - b_n)  (bid level 1 同理 ask)
  OFI = ΔV_b - ΔV_a
"""
import numpy as np
import pandas as pd


def calc_obi(bid_qty: float, ask_qty: float) -> float:
    return (bid_qty - ask_qty) / (bid_qty + ask_qty) if (bid_qty + ask_qty) else 0.0


def calc_microprice(bid_px: float, ask_px: float,
                    bid_qty: float, ask_qty: float) -> float:
    return (bid_px * ask_qty + ask_px * bid_qty) / (bid_qty + ask_qty)


def calc_ofi(bid_px_prev: float, bid_px_now: float,
             bid_qty_prev: float, bid_qty_now: float) -> float:
    """Cont 2021 OFI 单边公式（bid 端），ask 端对称"""
    if bid_px_now > bid_px_prev:
        return bid_qty_now
    elif bid_px_now < bid_px_prev:
        return -bid_qty_now
    else:
        return bid_qty_now - bid_qty_prev


def ofi_series(df: pd.DataFrame) -> pd.Series:
    """
    df: 含 ['bid_px', 'bid_qty', 'ask_px', 'ask_qty']
    返回每个时间点的 OFI
    """
    bid_v = (df['bid_qty'] - df['bid_qty'].shift(1)) \
              * np.sign(df['bid_px'] - df['bid_px'].shift(1)).fillna(0)
    ask_v = (df['ask_qty'] - df['ask_qty'].shift(1)) \
              * np.sign(df['ask_px'] - df['ask_px'].shift(1)).fillna(0)
    return bid_v - ask_v
```

## A.3 Avellaneda-Stoikov 做市报价

```python
"""
经典做市理论：Avellaneda & Stoikov (2008)
- 输入：S (中间价), σ (短期波动率), q (inventory 头寸), T (剩余时间), γ (风险厌恶)
- 输出：最优 reservation price r 与 half-spread δ
  r = S - q * γ * σ² * (T - t)
  δ = (γ * σ² * (T - t) + 2/γ * ln(1 + γ/κ)) / 2
  bid_quote = r - δ
  ask_quote = r + δ
- κ: order book depth parameter (≈ A * K / σ² 通常 1-10)
"""
import numpy as np


class AvellanedaStoikov:
    def __init__(self, gamma: float = 0.1, kappa: float = 1.5,
                 T: float = 1.0):
        self.gamma = gamma   # 风险厌恶
        self.kappa = kappa   # order book depth
        self.T = T           # 持仓周期

    def quotes(self, S: float, sigma: float, q: float, t: float = 0):
        tau = max(self.T - t, 1e-9)
        r = S - q * self.gamma * sigma**2 * tau
        delta = (self.gamma * sigma**2 * tau
                 + 2 / self.gamma * np.log(1 + self.gamma / self.kappa)) / 2
        bid = r - delta
        ask = r + delta
        return {"reservation": r, "half_spread": delta,
                "bid": bid, "ask": ask, "inventory": q}
```

## A.4 Queue Position 建模（执行概率）

```python
"""
Queue position: 你的限价单排在队列第几位
- 决定成交概率
- 论文：Cont, Stoikov, Talreja (2010)
- 简化：泊松到达假设
  P(exec) = 1 - exp(-λ * Δt)  其中 λ = arrival rate
  arrival rate 与价格档位的距离 d = (your_px - best_px)/σ 有关系
"""
import numpy as np


def exec_probability(distance_sigma: float, dt_sec: float,
                     base_rate: float = 1.0) -> float:
    """
    distance_sigma: 你的报价距 best price 的 σ 数（正值 = 不在 best）
    dt_sec: 持有时间
    base_rate: best 价的到达率（per sec）
    """
    if distance_sigma <= 0:
        lam = base_rate
    else:
        lam = base_rate * np.exp(-distance_sigma)  # 距离衰减
    return 1 - np.exp(-lam * dt_sec)


def expected_pnl(px: float, fill_prob: float, mid: float,
                 rebate: float = 0.0002, adverse: float = 0.0003) -> float:
    """
    做市单笔期望 PnL
    rebate: maker 返佣
    adverse: 成交后被吃损（adverse selection）的成本
    """
    fill_value = px - mid   # 卖单为 +，买单为 -
    return fill_prob * (fill_value + rebate - adverse) - (1 - fill_prob) * 0
```

## A.5 Spoofing / Layering 检测（监管合规）

```python
"""
检测异常撤单模式（CFTC 监管定义）
- Spoofing: 大单挂出后快速撤掉，方向上小单获利
- Layering: 多档同向大单，制造虚假深度
- 启发式特征：
  1) order cancel ratio > 0.95 且 单笔 > 5× median
  2) 取消时间 < 100ms
  3) 同一方向（bid/ask）连续 > 3 档累计 > 10× 平均
"""
import pandas as pd
import numpy as np


def detect_spoofing(orders: pd.DataFrame, cancel_window_ms: int = 100,
                    size_threshold: float = 5.0) -> pd.DataFrame:
    """
    orders: DataFrame 含 ['account', 'side', 'px', 'qty', 'placed_ts', 'cancel_ts']
    返回可疑订单
    """
    if orders.empty:
        return orders
    orders = orders.copy()
    orders["lifetime_ms"] = (orders["cancel_ts"] - orders["placed_ts"]).dt.total_seconds() * 1000
    orders["size_z"] = (orders["qty"] - orders["qty"].median()) / orders["qty"].std()

    suspicious = orders[
        (orders["lifetime_ms"] < cancel_window_ms) &
        (orders["size_z"] > size_threshold)
    ]
    return suspicious


def detect_layering(book_snapshots: pd.DataFrame, depth_levels: int = 5) -> pd.DataFrame:
    """
    book_snapshots: 不同时刻的 L2 快照
    检测同侧连续多档显著大单
    """
    if book_snapshots.empty:
        return book_snapshots
    book_snapshots = book_snapshots.copy()
    book_snapshots["bid_cum_qty"] = book_snapshots["bids"].apply(
        lambda b: sum(q for _, q in b[:depth_levels]))
    book_snapshots["ask_cum_qty"] = book_snapshots["asks"].apply(
        lambda a: sum(q for _, q in a[:depth_levels]))
    threshold = book_snapshots["bid_cum_qty"].mean() + 3 * book_snapshots["bid_cum_qty"].std()
    return book_snapshots[
        (book_snapshots["bid_cum_qty"] > threshold) |
        (book_snapshots["ask_cum_qty"] > threshold)
    ]
```

## A.6 事件驱动回测引擎（Order Book 模拟）

```python
"""
极简事件驱动回测 — 处理 limit / market 单
- 维护内部 order book
- 处理 partial fills, time priority
"""
from collections import deque
from dataclasses import dataclass, field
from typing import Optional
import heapq


@dataclass
class Order:
    ts: int
    side: str       # 'buy' or 'sell'
    type: str       # 'limit' or 'market'
    px: Optional[float]
    qty: float
    account: str = "user"


@dataclass
class Fill:
    ts: int
    px: float
    qty: float
    taker_side: str


class OrderBook:
    def __init__(self):
        self.bids = {}     # {px: deque of (ts, qty)}
        self.asks = {}
        self.best_bid = 0.0
        self.best_ask = float('inf')

    def add_limit(self, order: Order):
        book = self.bids if order.side == 'buy' else self.asks
        if order.px not in book:
            book[order.px] = deque()
        book[order.px].append((order.ts, order.qty))
        self._update_best()

    def _update_best(self):
        self.best_bid = max(self.bids.keys()) if self.bids else 0.0
        self.best_ask = min(self.asks.keys()) if self.asks else float('inf')

    def process_market(self, order: Order) -> list[Fill]:
        fills = []
        remaining = order.qty
        book = self.asks if order.side == 'buy' else self.bids
        while remaining > 0 and book:
            best_px = min(book.keys()) if order.side == 'buy' else max(book.keys())
            queue = book[best_px]
            while queue and remaining > 0:
                ts, qty = queue.popleft()
                fill_qty = min(remaining, qty)
                fills.append(Fill(ts=order.ts, px=best_px, qty=fill_qty,
                                  taker_side=order.side))
                remaining -= fill_qty
                if qty > fill_qty:
                    queue.appendleft((ts, qty - fill_qty))
            if not queue:
                del book[best_px]
        self._update_best()
        return fills
```

## A.7 Almgren-Chriss 滑点模型（大单拆单）

```python
"""
Almgren-Chriss 1999 模型：交易冲击 = temporary + permanent
- 用于估算 TWAP/VWAP 实际滑点
- T: 持仓周期（秒）
- X: 总交易量
- σ: 标的波动率
- η: 临时冲击系数
- γ: 永久冲击系数
- 最优拆单: 均匀分布 (RiskNeutral) 或 U-shape (RiskAverse)
"""
import numpy as np


def almgren_chriss_slippage(X: float, T: float, sigma: float,
                            eta: float = 2.5e-7, gamma: float = 2.5e-7,
                            risk_aversion: float = 1e-4) -> dict:
    """
    返回：临时冲击成本 + 永久冲击成本 + 最优分片数
    """
    tau = T / 1.0
    perm_cost = gamma * X
    temp_cost = eta * X**2 / T
    # 风险调整
    risk_cost = risk_aversion * sigma**2 * T * X**2 / 3
    # 最优 U-shape 拆单数（K=1 均匀）
    K_optimal = max(1, int(np.sqrt(risk_aversion * sigma**2 * T / eta)))
    return {
        "permanent": perm_cost,
        "temporary": temp_cost,
        "risk_adjusted": risk_cost,
        "total_slippage_bps": (perm_cost + temp_cost + risk_cost) / X * 10000,
        "optimal_slices": K_optimal,
    }
```

```python
"""
完整做市机器人骨架（生产参考）

class MarketMakerBot:
    def __init__(self, symbol: str = "BTC/USDT", gamma: float = 0.05,
                 kappa: float = 2.0, target_inventory: float = 0.0,
                 max_inventory: float = 0.5):

        self.symbol = symbol
        self.mm = AvellanedaStoikov(gamma=gamma, kappa=kappa, T=0.005)
        self.target_inventory = target_inventory
        self.max_inventory = max_inventory
        self.inventory = 0.0

        mid = (best_bid + best_ask) / 2

        micro = calc_microprice(best_bid, best_ask, bid_qty, ask_qty)

            quotes["bid"] -= 10   # 主动买
            quotes["ask"] += 10   # 主动卖

        quotes["bid"] += obi * 5
        quotes["ask"] += obi * 5

            await self.exchange.create_limit_buy_order(self.symbol, size, bid)
            await self.exchange.create_limit_sell_order(self.symbol, size, ask)

            print(f"order err: {e}")

            ob = await self.exchange.fetch_order_book(self.symbol, limit=20)

                ob['bids'][0][0], ob['asks'][0][0],
                ob['bids'][0][1], ob['asks'][0][1])
            await asyncio.sleep(0.5)

    async def close(self):
        await self.exchange.close()
```
